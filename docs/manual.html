1  Overview

fishio.2 reads water temperature from one or more DS18B20 sensors on a shared OneWire bus, maps each sensor to a friendly tank name, and:

  • shows live readings on a 240×240 round GC9A01 LCD (carousel or list layout);
  • publishes each tank's temperature to MQTT;
  • publishes device health (heap, RSSI, uptime) every 60 s;
  • raises a buzzer + Telegram alert when a tank goes out of range or a sensor disconnects;
  • supports password-protected over-the-air (OTA) firmware updates.

It is built with PlatformIO on the Arduino framework. Firmware is organized into single-purpose manager classes: WifiManager, MqttManager, TempSensorManager, DisplayManager, Buzzer, TelegramAlert, SystemInfoManager, and OtaManager.

3  Pin layout

All display and sensor signals are deliberately on the same DevKit header side (GPIO 4, 5, 16, 17, 18, 23) for tidy wiring. The buzzer is on GPIO26.

Active connections

FunctionESP32 GPIONotes
Display SCL (SPI clock)GPIO18VSPI SCLK
Display SDA (SPI data)GPIO23VSPI MOSI
Display DCGPIO16Data/command
Display CSGPIO4Chip select
Display RSTGPIO17Reset
DS18B20 DATAGPIO5+ 4.7 kΩ pull-up to 3V3 (one per bus)
Buzzer +GPIO26LEDC PWM

Display VCC and DS18B20 VDD go to 3V3; all grounds to GND. The module's physical header order is VCC, GND, SCL, SDA, DC, CS, RST.

GPIO5 is a strapping pin, but the OneWire 4.7 kΩ pull-up holds it HIGH at boot — exactly the level the strap requires — so it is safe for the sensor data line.

5  Display dashboard

The round 240×240 panel has two layouts, chosen by DISPLAY_MODE in Config.h. Both color-code status: green = in range red = out of range yellow = sensor error.

Figure 2 — Carousel (status ring + page dots) vs. List (rows with status dots).

On boot the screen shows a startup message, then the WiFi IP, then the dashboard. With no sensors on the bus it shows “No sensors.” Carousel dwell time is set by CAROUSEL_INTERVAL_MS (default 3000 ms).

6  Configuration

Secrets (local, git-ignored)

Copy src/Secrets.h.example to src/Secrets.h and fill in:

#define WIFI_SSID        "your-ssid"
#define WIFI_PASSWORD    "your-password"
#define TELEGRAM_BOT_TOKEN "123456789:AA..."
#define TELEGRAM_CHAT_ID   "123456789"
#define OTA_PASSWORD     "change-me"
// Optional: #define TELEGRAM_ROOT_CA  R"EOF( ...PEM... )EOF"  to verify TLS

Main settings (src/Config.h)

SettingPurpose
MQTT_SERVER / _PORTBroker address and port
MQTT_CLIENT_IDUnique per device (also used in the health topic)
TEMP_LOW / TEMP_HIGHAlert thresholds in °C (default 20 / 28)
DISPLAY_MODEDISPLAY_CAROUSEL or DISPLAY_LIST
CAROUSEL_INTERVAL_MSCarousel dwell time per tank
TFT_SPI_SPEEDDisplay SPI clock (default 8 MHz)
TANKS[]Sensor-ID → tank-name map

To onboard a tank, add its DS18B20 64-bit address and a name to TANKS[]. The address is printed to serial on boot (Sensor N initialized (ID: …)).

7  Build & flash

pio run                     # build the main firmware
pio run --target upload     # flash over USB
pio device monitor          # serial monitor @ 115200

After the first USB flash, push updates over the air (password-protected ArduinoOTA) — the device appears as a network port in PlatformIO.

Display diagnostic

A standalone diag environment cycles color fills, shapes, and text directly on the panel (bypassing the app) to isolate wiring vs. SPI-speed problems:

pio run -e diag -t upload
pio device monitor

8  MQTT & alerts

TopicPayload
temp/aquarium/<sensorId>Temperature °C, e.g. 25.43
system/<clientId>/statusJSON: chipId, heapFree, flashSize, wifiRSSI, uptimeSec

Sensors are read every 10 s (one bus-wide conversion per cycle). Device health publishes every 60 s.

Alerts. When a tank reads below TEMP_LOW, above TEMP_HIGH, or returns a disconnect, the buzzer sounds and a Telegram message is sent. Alerts are debounced per sensor: one message on entering a fault, one “back to normal” on recovery — no spam.

Telegram uses HTTPS POST. By default the certificate is not verified (a warning is logged). Define TELEGRAM_ROOT_CA in Secrets.h to pin it.

9  Future expansion

The board is provisioned for growth without re-wiring the core:

  • Relays / solenoids (heaters, lights, pumps) — GPIO27/32/33, via opto-isolated relay/SSR boards with a 10 kΩ pull-down each so nothing switches on at boot.
  • Analog probes (pH, TDS, ORP, level) — ADC1 pins GPIO36/39/34/35, or an ADS1115 on the I²C bus for clean 16-bit readings.
  • I²C bus — GPIO21 (SDA) / GPIO22 (SCL), free for ADS1115, an MCP23017 relay bank, an RTC, etc.

Actuator loads always get their own power supply with a common ground — the ESP32 only drives the relay/SSR logic input. DC solenoids need a flyback diode.

10  Troubleshooting

SymptomLikely cause & fix
Display backlight on but blank SPI wiring or clock. Verify SCL(18)/SDA(23) aren't swapped and RST(17)/DC(16)/CS(4) are solid. Lower TFT_SPI_SPEED (jumper wires often can't do high clocks). Flash the diag build to confirm the panel responds.
Sensor enumerates but reads −127 °C Missing/weak pull-up. Ensure a 4.7 kΩ resistor sits between the GPIO5 data line and 3V3 (not the old pin). The internal pull-up alone reads the address but fails conversions.
“No sensors” on screen Nothing on the OneWire bus — check DATA→GPIO5, VDD→3V3, GND, and the pull-up.
Telegram alert fails (HTTP 401) Bad bot token / chat ID in Secrets.h. A 200 response means success.
Won't boot after wiring a new pin You may have used a strapping pin (0/2/12/15) held at the wrong level. Move the signal to a free GPIO.
Device freezes ~50 s periodically Old behavior on an unreachable MQTT broker — current firmware does a single non-blocking connect attempt per cycle, so update if you see this.
